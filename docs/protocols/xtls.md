# 🚗 XTLS Reality + Vision (Umbrella Edition)

XTLS — это эволюция протокола VLESS, предлагающая лучшую на сегодня маскировку под стандартный HTTPS трафик (TCP).

Данная реализация: SOCKS5 прокси поверх TCP:443 с [Reality](https://github.com/XTLS/REALITY) handshake и yamux мультиплексированием.

## Конфигурация

Для активации этого режима используйте `protocol: xtls` в конфигурационных файлах.

### Сервер (config.yaml)
```yaml
protocol: "xtls"
port: "443"             # Порт прослушивания
dest: "samsung.com:443" # Сайт-донор для Reality (куда перенаправлять "чужих")

xtls:
  private-key: "..."    # x25519 приватный ключ (base64, 32 байта)
  short-id: "..."       # Reality Short ID (hex, до 16 символов)
  server-names: "..."   # Допустимые SNI через запятую (если пусто, берется из dest)
```

### Клиент (config.yaml)
```yaml
protocol: "xtls"
server: "your_vps_ip:443"
sni: "samsung.com"      # Должен совпадать с одним из SNI сервера
listen: "0.0.0.0:1080"
udp: true               # Поддержка UDP ASSOCIATE
decoy-traffic: true     # Глобальная маскировка (внешние HEAD-запросы)

xtls:
  public-key: "..."     # Публичный ключ сервера (base64)
  short-id: "..."       # Short ID сервера (hex)
  sessions-num: 5       # Количество yamux-сессий в пуле
  connections-time-out: 60 # Таймаут бездействия стрима (сек)
```

## Технологии маскировки

### 1. Reality (Handshake Stealth)
Технология Reality позволяет вашему серверу "красть личность" у любого популярного зарубежного сайта (например, `samsung.com` или `microsoft.com`).
*   **Как это работает:** При попытке зайти на ваш сервер по порту 443, он проксирует TLS-рукопожатие от реального выбранного сайта. 
*   **Результат:** Провайдер видит валидный сертификат и легитимные TLS-отпечатки. Ваш сервер Umbrella не выдает себя, пока клиент не отправит секретный ключ в поле `SessionID`.

---

Аутентификация встроена в TLS ClientHello, а не в прикладные данные:

1. Клиент запускает `BuildHandshakeState()` — ClientHello строится в памяти без отправки.
2. Клиент извлекает эфемерный x25519-ключ из KeyShare и вычисляет `sharedSecret = X25519(clientEphPriv, serverStaticPub)`.
3. `authKey = HKDF-SHA256(ikm=sharedSecret, salt=random[:20], info="REALITY")`.
4. Plaintext (16 байт) `[ver=0 | zeros | unix_time(4) | shortId(8)]` шифруется `AES-256-GCM(authKey, nonce=random[20:32])`, результат (32 байта) записывается в поле `SessionId`.
5. `MarshalClientHello()` → `Handshake()`.

Сервер (`xtls/reality`) делает то же самое независимо: если расшифровка `SessionId` валидна — соединение аутентифицировано. Иначе — прозрачный прокси к `dest` (github.com). Сервер никогда не показывает свой собственный сертификат.

### 2. Umbrella Vision
**Задача.** 

Когда приложение открывает HTTPS-соединение через туннель, его TLS ClientHello и весь inner handshake проходят в зашифрованном виде через outer Reality TLS. DPI не читает содержимое, но видит **статистический паттерн размеров первых TLS-записей** (inner ClientHello ≈ 300–500 байт, затем характерная последовательность record'ов) — это детектируемый признак TLS-in-TLS.

В целом все пакеты приложений могут иметь статистические сигнатуры длин.

**Решение.**

Все соединения оборачиваются в Vision-поток, а все пакеты в Vision-фреймы
  * Генерируется фейковый TLS-заголовок
  * Реальные TLS-заголовоки и данные приложения упаковываются в payload
  * **Random Padding:** В каждое сообщение добавляется от 0 до 255 байт случайного мусора.
  * Это делает длину каждого пакета уникальной и непредсказуемой, ломая любые статистические сигнатуры длин пакетов.
  * Засчет этих решений туннель корректно пропускает нестандартные протоколы на порту 443 (например, MTProto у Telegram) при этом продолжая маскировать TLS-заголовки + основные данные используя Random Padding. Не нужны дополнительные настройки. Все происходит прозрачно

### 3. Мультиплексирование (yamux)

Все SOCKS5-потоки идут внутри одного TLS-соединения через yamux. Нет лавины отдельных TLS-handshake'ов при многопоточной загрузке.

### 4. Ротация сессии

Клиент ротирует TLS-соединение каждые **3–15 минут** случайным образом (`crypto/rand`). Активные потоки yamux дожидаются естественного завершения. Следующий SOCKS5-запрос прозрачно открывает новое соединение с новым handshake.

## Преимущества и недостатки перед Xray (VLESS+REALITY+XTLS)

Оригинальный XTLS нацелен на максимальную производительность (снятие обертки) и строгую маршрутизацию, ради чего жертвует гибкостью. Umbrella использует другой подход, ориентируясь на отсутствие конфигурации и безотказность.

**Преимущества Umbrella:**

1. **Ничего не ломает (Универсальность):** Клиент применяет Vision ко всему трафику, при этом корректно заворачивает и обрабатывает поток данных нестандартных протоколов на порту 443 (например, MTProto у Telegram), не отключая для них маскировку. Клиенту Xray с `xtls-rprx-vision` потребовались бы сложные правила маршрутизации, иначе не-TLS трафик на настроенном порту обрывался бы для защиты от детектирования (active probing).
2. **Нулевая настройка (Zero-config):** Администратору или пользователю не нужно думать о директивах `flow`. Протокол подстраивается сам для каждого нового потока на лету.
3. **Безопасность мультиплексирования:** Через единый внешний TCP-туннель (Reality) внутри `yamux` проходят сотни внутренних SOCKS5-запросов. DPI никогда не узнает количество открытых подключений. В оригинальном Xray каждый сплайсированный поток порождает отдельное внешнее подключение.
4. **Random Padding:** продолжает ломать статистические сигнатуры длин пакетов для всего трафика, а не только для TLS ClientHello.

**Недостатки Umbrella:**

1. **Оверхед процессора (Двойное шифрование):** Xray XTLS делает "сплайсинг" — отключает слой VLESS на стадии Application Data и отправляет внутренний трафик наружу "как есть" (поскольку он уже зашифрован внутренним HTTPS). Из-за `yamux` Umbrella вынуждена всегда шифровать трафик вторым слоем (внешним сервером Reality). Это создает небольшую (1-2%) лишнюю нагрузку на процессор VPS/клиента, что может быть заметно на гигабитных скоростях роутеров, но незаметно в повседневном использовании.

## Когда использовать?
*   Основной режим для работы и серфинга.
*   Если UDP заблокирован или сильно ограничен.
*   В сетях с жестким "белым списком" доменов (Reality поможет прикинуться сайтом из списка).
